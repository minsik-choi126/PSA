# Policy-Shift-Guided Spectral Alignment (PSA)

Self-contained reference implementation of the 3-expert merge method:

> column-side activation magnitudes weighted by **|О”logp|** (abs + soft per-token
> weighting), corrected by per-column W-norm, used inside an energy-truncated
> column-weighted SVD with cross-expert polar alignment and per-expert
> Frobenius renorm.

Two model variants ship out of the box:

| variant       | base                    | 3 experts                                          | per_query bundled? |
|---------------|-------------------------|----------------------------------------------------|--------------------|
| **Qwen2.5-7B**  | `Qwen/Qwen2.5-7B-Instruct` | coding В· tool В· memory                          | вњ“ (data/qwen2.5_7b/per_query/) |
| **Qwen3-1.7B**  | local `Qwen3-1.7B-base`    | if_rl В· math В· lucy                              | вЂ“ (auto-generated by run script on first invocation) |

## Method

For each Linear layer в„“, expert i в€€ {3 RL experts}:

### 1. Column-side activation magnitude (abs + soft)

For every answer-token position t in the saved per-expert rollout:

```
О”logp[t] = log p_E(y_t)  -  log p_base(y_t)
w[t]     = |О”logp[t]| ^ О±              (О±=1.0 default в†’ linear)
```

A forward hook on every `nn.Linear` accumulates `ОЈ_t w[t] В· |x[t,c]|` directly
in GPU memory (no per-call clone). After processing all prompts:

```
W_col[i, в„“, c] = ( ОЈ_t w[t] В· |x_i,в„“(t)[c]| ) / ОЈ_t w[t]
```

No top-K threshold; every token contributes proportional to its own |О”logp|.

### 2. W-norm correction

```
П‰_col[i, в„“, c] = вЂ–W_expert_i,в„“[:, c]вЂ–в‚‚  В·  W_col[i, в„“, c]
```

Restores the |WВ·x| symmetry between row and column sides.

### 3. Column-weighted SVD truncation

```
П„_i      = W_expert_i  -  W_base
Y_i      = П„_i  В·  D_c^{1/2}             D_c = diag(П‰_col[i, в„“, :])
SVD(Y_i) в†’ U_y, Пѓ_y, V_y
K_i      = smallest k s.t. ОЈ Пѓ_y[:k]ВІ в‰Ґ energy В· ОЈ Пѓ_yВІ    (energy=0.90)
П„_i^(K)  = (U_y[:,:K] В· diag(Пѓ_y[:K]) В· V_y[:K,:])  В·  D_c^{-1/2}
```

### 4. Polar alignment via QR (avoid second full-rank SVD)

Polar alignment needs original-space orthonormal U, V. We avoid a second
full SVD by reusing the weighted-space SVD result with a tiny KГ—K SVD:

```
M_right_i = V_y[:K] В· D_c^{-1/2}                    # (K Г— d_in)
QR(M_right_i^T) в†’ Q_r_i в€€ R^{d_in Г— K}, R_r_i в€€ R^{K Г— K}
middle_i  = diag(Пѓ_y[:K]) В· R_r_i^T                  # (K Г— K) вЂ” small
SVD(middle_i) в†’ U_m_i, S_m_i, V_m_i
U_k_i  = U_y[:K] В· U_m_i                              # orthonormal d_out Г— K
V_k_i  = Q_r_i В· V_m_i^T                              # orthonormal d_in  Г— K
S_k_i  = S_m_i
```

For lm_head (152064 Г— 3584, K~1k) this drops the dominant cost from
в‰€2 Г—10В№ВІ ops в†’ в‰€1.5 Г—10В№В№ ops per expert per layer (в‰€10Г— speedup of resvd).

### 5. Cross-expert polar + per-expert renorm

```
U_hat        = polar_factor( cat(U_k_1 | U_k_2 | U_k_3) )
V_hat        = polar_factor( cat(V_k_1 | V_k_2 | V_k_3) )
П„_aligned_i  = U_hat[blocks_i] В· diag(S_k_i) В· V_hat[blocks_i]^T
О±_i          = вЂ–П„_iвЂ– / вЂ–П„_aligned_iвЂ–
merged_П„_в„“   = ОЈ_i О±_i В· П„_aligned_i
W_merged_в„“   = W_base_в„“ + merged_П„_в„“
```

`polar_factor(X) = U @ Vh` of `SVD(X)`, the nearest orthogonal matrix.

### 1D / non-2D parameters
Uniform task-vector mean: `Оё_merged = Оё_base + (1/N) ОЈ (Оё_i - Оё_base)`.

## Per-query data

**Schema** (npz):

```
full_tokens     (ОЈ full_seq_lens,) int32     # flattened prompt+answer tokens
prompt_lens     (N,)               int32
full_seq_lens   (N,)               int32
seq_lens        (N,)               int32     # answer length per query
expert_names    (E,)               str       # ordered as in --experts
base_lp         (T,)               float32   # log p_base over answer tokens
expert_lp       (E, T)             float32   # log p_E    over answer tokens

```

**Bundled вЂ” Qwen2.5-7B**: produced from the 3 experts' own training corpora
(in-domain calibration). Per-task generation hyperparameters:

| task   | source                  | T (gen)  | max_new_tokens |
|--------|-------------------------|----------|----------------|
| coding | CodeContests-train      | 0.8      | 2048           |
| tool   | ToolRL-train            | 0 greedy | 256            |
| memory | HotpotQA-train-32K      | 0.7      | 256            |

**Bundled вЂ” Qwen3-1.7B prompts** (npz auto-generated on first run):

| task  | prompt source                                                | size |
|-------|--------------------------------------------------------------|------|
| if_rl | local IFEval verl-format prompts                             | 128  |
| math  | local math RL training set                                   | 128  |
| lucy  | `Menlo/sft-lucy-musique-data-corrected-with-completion` train | 128  |

`gen_per_query.py` will roll out from the base model (T=0.7, 256 new tokens
by default) and teacher-force-score under base + each expert. About 10-15 min
on a single GPU for the 1.7B variant.

If you swap experts, update `EXPERT_*` env vars; for new tasks, drop a
`<name>.jsonl` (chat-format prompts) into the corresponding `prompts/` dir
and re-run.

## Troubleshooting

- **OOM at extract_w**: long-context tasks (e.g. memory_rl on 25k-token
  rollouts) use a lot of VRAM. Reduce `--max_prompts` or run with `DP=1`.
- **Step 2 (Wnorm) is CPU-bound**: it loads each expert's full state_dict to
  compute per-column вЂ–W[:,c]вЂ–в‚‚. Plan for в‰€60 GB RAM headroom for 7B (3 Г— ~15 GB
  fp32 state-dicts), в‰€15 GB for 1.7B.
- **`tokenizer.chat_template is not set`**: some bases ship without a
  default chat template. Either provide `chat_template.jinja` in the model
  dir or convert prompts to plain text and adjust `gen_per_query.py`.
- **Different total token count between in-domain and OOD calibration runs**:
  expected вЂ” different prompts/lengths produce different ОЈw. The W npz is
  internally normalised by ОЈw before merge.
